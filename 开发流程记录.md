# uniapp_wallpaper 开发流程全记录

## 项目背景

从零搭建壁纸小程序：将原有的第三方 API 依赖（`tea.qingnian8.com`）完全替换为自建 uniCloud 后端，实现数据自主可控。

**目标架构：** 前端 → `uniCloud.importObject('wallpaper')` → 云对象 → 自建数据库

## 阶段一：基础搭建

### 1.1 数据库表设计

设计 6 张表，覆盖壁纸小程序的全部数据需求：

```
wallpaper-classify (分类)
    │ 1:N
    ▼
wallpaper-list (壁纸详情，核心表)
    │ 1:N
    ├── wallpaper-user-score (评分记录)
    └── wallpaper-download-record (下载记录)

wallpaper-banner (轮播图) —— 独立
wallpaper-news (公告) —— 独立
```

**核心设计决策：**
- 壁纸表（wallpaper-list）设置 `status` 字段：0=待审核 / 1=已发布 / 2=已下架，支持内容审核流程
- 评分表和下载表使用 `deviceId`（设备 ID）暂代用户标识，后续可无缝迁移到 uni-id 的 `openid`
- 评分表设置**唯一联合索引** `{wallId, deviceId}`，从数据库层面防止重复评分
- 壁纸表建立复合索引 `{classid, status}` 和 `{status, create_time}`，优化分类筛选和内容管理查询

### 1.2 Schema 权限配置

| 表类型 | read | create | update | delete |
|--------|------|--------|--------|--------|
| 内容表（banner/classify/list/news） | ✅ | ❌ | ❌ | ❌ |
| 用户表（user-score/download-record） | ✅ | ✅ | ❌ | ❌ |

前端用户只能读取内容、写入评分和下载记录。后台所有操作通过云对象完成，云对象使用传统数据库模式绕过 Schema 限制。

### 1.3 关键技术选型

- **传统数据库模式（db + dbCmd）** 而非 JQL 模式
  - JQL 受 Schema 限制（如 `create:false` 会阻止云对象写入）
  - JQL 不支持 `dbCmd.inc()` / `aggregate()` / `lookup()` 等高级操作
  - 云对象作为服务端代码，理应拥有完整数据库权限

## 阶段二：云对象开发

### 2.1 _before 预处理

```javascript
_before: function () {
    const clientInfo = this.getClientInfo()
    this.deviceId = clientInfo.deviceId || 'anonymous'
}
```

每个方法执行前自动提取设备 ID 作为用户标识，所有涉及用户的操作用 `this.deviceId`。

**为什么不引入 uni-id？** 当前阶段只做 CRUD + 展示，用设备 ID 足够。后续只需将 `deviceId` 替换为 `clientInfo.uid` 即可完成迁移。

### 2.2 前端调用方法（12 个）

#### getRandomWall — 随机推荐

核心思路：先 count 总数，再随机 skip 一段偏移量，取 9 条。比 `aggregate().sample()` 更可靠（支付宝云不支持 sample）。

```javascript
const total = await db.collection('wallpaper-list').where({status:1}).count()
const skip = Math.floor(Math.random() * (total - 9))
const result = await db.collection('wallpaper-list')
    .where({status:1}).skip(skip).limit(9).get()
```

#### setupScore — 评分（核心业务逻辑）

两步操作，先写评分记录（利用唯一索引防重复），再原子更新平均分：

```javascript
// 1. 写入评分记录 → 唯一索引冲突 = 已评分过
await db.collection('wallpaper-user-score').add({wallId, deviceId, userScore, ...})

// 2. 重新计算平均分
const newScore = (oldScore * oldCount + userScore) / (oldCount + 1)
await db.collection('wallpaper-list').where({_id: wallId})
    .update({score: newScore, scoreCount: newCount})
```

#### downloadWall — 下载计数

```javascript
await db.collection('wallpaper-download-record').add({wallId, deviceId, ...})
await db.collection('wallpaper-list').where({_id: wallId})
    .update({downloadCount: dbCmd.inc(1)})  // 原子自增
```

#### getUserWallList — 用户历史（联表查询）

使用 `aggregate().lookup()` 联查壁纸详情，再展平结果。比多次查询高效。

#### searchWall — 搜索（关键坑点）

**问题：** 支付宝云不支持 `dbCmd.regex()`，运行时抛出 `TypeError: dbCmd.regex is not a function`

**解决方案：** 拉取已发布壁纸（最多 200 条），在 JavaScript 中用 `String.includes()` 做模糊匹配，最后在内存中分页。数据量在几百条时性能完全可接受。

### 2.3 后台管理方法（20 个）

覆盖轮播图、分类、壁纸、公告的完整 CRUD，加上批量导入和壁纸统计。

#### adminUpload — 供爬虫上传图片

```javascript
async adminUpload(data) {
    const result = await uniCloud.uploadFile({
        cloudPath: `wallpaper/${data.fileName}`,
        fileContent: Buffer.from(data.base64, 'base64')
    })
    const urlResult = await uniCloud.getTempFileURL({fileList: [result.fileID]})
    return { errCode: 0, data: { fileID: result.fileID, cloudURL: ... } }
}
```

#### adminBatchImport — 供爬虫批量导入

每次最多 50 条壁纸批量写入，爬虫采集后直接调用此方法入库。

#### adminDeleteWall — 级联删除

删除壁纸时同步删除关联的评分记录和下载记录，保持数据一致性。

#### adminDeleteClassify — 安全检查

删除分类前检查是否还有关联壁纸，防止出现孤儿数据。

## 阶段四：HTTP URL 化调试

### 4.1 核心问题

云对象开启 URL 化后，Python 爬虫通过 HTTP POST 调用时发现：**POST JSON body 的参数无法被云对象方法接收**，只有 query string 参数能正常传递。

### 4.2 原因分析

uniCloud URL 化机制对 POST body 的处理方式与 query string 不同。通过 `this.getHttpInfo()` 可以获取原始 HTTP body，但需要手动解析。

### 4.3 解决方案

在云对象中添加 `mergeData()` 辅助函数：

```javascript
function mergeData(data) {
    const httpInfo = this.getHttpInfo ? this.getHttpInfo() : null
    let bodyData = {}
    if (httpInfo && httpInfo.body) {
        try {
            bodyData = typeof httpInfo.body === 'string'
                ? JSON.parse(httpInfo.body) : httpInfo.body
        } catch (e) {}
    }
    return Object.assign({}, bodyData, data)
}
```

每个接收 data 参数的方法开头调用 `data = mergeData.call(this, data)`，合并 POST body 和 query string 两个来源的参数。

### 4.4 部署验证

修改云对象后需在 HBuilderX 中上传部署到 uniCloud。部署后通过 Python 脚本验证 POST JSON body 能正常传递参数。

## 附录：uni-app / uniCloud 核心知识体系

> 以下内容来自开发过程中积累的学习笔记（`uniapp开发.txt` / `开发uniCloud项目.txt`），涵盖 uni-app 项目搭建到 uniCloud 高级用法的完整知识脉络。

### A.1 项目搭建流程

```
HBuilderX 新建 uni-app 项目
  ├── 勾选"启用 uniCloud"
  ├── 选择服务供应商（支付宝云 / 阿里云 / 腾讯云）
  └── 创建完成
        ├── 右键项目 → 关联 uniCloud 服务空间
        ├── uniCloud/cloudfunctions 目录右键 → 新建云函数/云对象
        └── 编写云对象方法 → 上传部署
```

### A.2 前后端分离架构

在 uniCloud 体系中，云函数/云对象承担**后端角色**：
- 客户端不直接操作数据库
- 客户端通过 `uniCloud.importObject('name')` 导入云对象
- 调用云对象方法获取数据，方法返回 `{errCode: 0, data: ...}` 统一格式
- 云对象在服务端执行，拥有完整数据库权限

### A.3 JQL vs 传统数据库模式

| 特性 | JQL | 传统模式（db） |
|------|-----|--------------|
| Schema 权限约束 | 受限制 | 不受限制 |
| dbCmd.inc() 原子操作 | ❌ | ✅ |
| aggregate().lookup() | 部分支持 | ✅ |
| 更新数组指定元素 | ❌ | ✅ |
| dbCmd.regex() 正则 | 视平台 | 视平台（支付宝云❌） |
| `$cloudEnv_uid` 环境变量 | ✅（依赖 uni-id） | ❌ |
| 适用场景 | 客户端简单查询 | 服务端复杂业务 |

**本项目选择传统模式**，因为云对象是服务端代码，需要 `inc()`、`aggregate()` 等高级操作能力。

### A.4 uniCloud-db 客户端查询组件

```vue
<!-- 联表查询：多个 collection 用逗号分隔 -->
<unicloud-db
  collection="demo-wallpaper,demo-classify"
  v-slot:default="{data, error, loading, hasMore}"
  field="name, picurl, createTime"

  <!-- data: 查询结果  error: 错误信息  loading: 加载状态  hasMore: 是否有更多 -->
</unicloud-db>
```

也可用 `colList` 数组代替逗号分隔。受 Schema 权限约束，无法执行 update/delete。

### A.5 云函数生命周期

| 钩子 | 类型 | 说明 |
|------|------|------|
| `_before` | 前置处理器 | 类似中间件/拦截器，用于 token 验证、提取用户信息 |
| `_after(error, result)` | 后置处理器 | 统一处理返回结果、日志记录。有 `_after` 时**必须 return 结果** |
| `_timing` | 定时触发器 | 在 `package.json` → `cloudfunction-config` 中配置 cron 表达式 |

**云对象特有 API：**
- `this.getClientInfo()` → 获取客户端平台、设备 ID、用户 uid（接入 uni-id 后）
- `this.getHttpInfo()` → URL 化时获取 HTTP method、headers、body

### A.6 URL 化（HTTP API）

**开启：** uniCloud 控制台 → 云函数/云对象 → 详情 → 开启 URL 化 → 获得长域名

**调用方式：** `POST https://域名/对象名/方法名`
- 参数可通过 query string 或 POST body 传递
- body 参数需在云对象中手动从 `this.getHttpInfo().body` 解析
- URL 有长度限制，大量数据必须走 body

### A.7 云存储核心操作

| 操作 | 客户端 API | 云函数 API |
|------|-----------|-----------|
| 选择图片 | `uni.chooseImage()` | — |
| 上传文件 | `uniCloud.uploadFile({cloudPath, fileContent})` | 同左 |
| 获取临时 URL | `uniCloud.getTempFileURL({fileList})` | 同左 |
| 删除文件 | `uniCloud.deleteFile({fileList})` | 同左 |

**关键点：**
- 推荐**客户端直传**，不经云函数中转，效率更高
- `uni-file-picker` 组件无法实现点击删除同步删除
- 公共权限文件永久有效，无需 `getTempFileURL`；私有权限文件有时效，需转换
- 可用 `dayjs().format("YYYYMMDD") + "/" + Date.now() + ".jpg"` 组织云存储路径
- 支付宝云的云存储不支持直接删除，需要通过云对象调用 `deleteFile`

### A.8 uni-id 用户体系要点

**配置：** `uni_modules/uni-config-center/uniCloud-alipay/uni-id/config.json`
- `passwordStrength`：密码强度规则
- `tokenSecret` / `requestAuthSecret`：Token 加密密钥
- `tokenExpiresIn`：登录状态有效期

**登录方式：**
- 微信小程序快捷登录（需条件编译 `#ifdef MP-WEIXIN`）
- 账号密码登录
- 短信验证码登录（需开通 SMS 短信服务 + 配置模板）

**JQL 环境变量：** `$cloudEnv_uid` 在 JQL 语句中获取当前用户 uid

**当前项目状态：** 用 `deviceId` 暂代用户标识，后续迁移路径为 `deviceId` → `openid`（`clientInfo.uid`）

### A.9 分页加载

页面 `onReachBottom` 生命周期 + `pageNum++` + 追加数据。本项目 classlist.vue 和 search.vue 使用此模式。

### A.10 图片处理

云存储支持图片缩放，路径后加参数可实现缩略图。本项目使用 Python Pillow 在采集阶段预处理生成 300px 宽 webp 缩略图。
